# ShortStoryMaker

ShortStoryMaker は、Codex CLI を使って短編小説を自動生成する実験プログラムです。

単に「AI に小説を書かせる」だけではなく、先に歴史上の出来事を調査し、そこから人物・状況・対立・情景・小道具などの物語素材を抽出し、それらをもとにプロット、本文、挿絵、HTML ページまで生成することを試みています。

このプログラムは、Codex CLI を「プログラムから呼べる AI 作業員」として扱うノウハウを実践したサンプルでもあります。

## 何をするプログラムか

実行すると、おおよそ次の流れで処理します。

1. ランダムな年月を選ぶ
2. その月に世界で起きた出来事を Codex に調査させる
3. 調査結果から、短編小説に使えそうな物語素材を抽出させる
4. 物語素材をもとにプロット案を複数作らせる
5. プロットと素材を組み合わせて短編小説を書かせる
6. 小説の冒頭に置く挿絵を生成させる
7. 小説本文と挿絵を表示する HTML ページを作らせる
8. 一覧ページ用の JavaScript ファイルを更新する

生成されたデータは、プロット、素材、小説、挿絵、HTML に分けて保存されます。

## 発想

LLM にそのまま「短編小説を書いて」と頼むこともできます。

しかし、それだけだと話の背景が薄くなったり、どこかで見たような設定になったりしがちです。そこでこのプログラムでは、まず歴史上の出来事を調べ、その出来事の周辺にいたかもしれない一般人の生活や葛藤を材料にして、物語を豊かにしようとしました。

狙いは、歴史を解説する小説を書くことではありません。

史実を一度、

- 人物
- 状況
- 対立
- 情景
- 小道具
- 物語の核

といった創作部品に変換し、それを小説本文へ食わせることです。

## Codex CLI の使い方

このプログラムの中心にあるのは、Codex CLI を非対話形式で呼び出す仕組みです。

Codex CLI は、ターミナルから対話的に使うだけでなく、次のように `exec` で一回限りの作業として実行できます。

```bat
codex -C "C:\temp\MyWork" exec ^
    --skip-git-repo-check ^
    --sandbox workspace-write ^
    "Prompt.md を読み、その内容を今回の作業指示として実行してください。"
```

この形にすると、Codex は指定された作業ディレクトリで `Prompt.md` を読み、その指示に従ってファイルを読み書きします。

### このオプションになった経緯

最初はもっと単純に、次のような形で実行していました。

```bat
codex -C "C:\temp\_SSM" exec "Prompt.md を読み、その内容を今回の作業指示として実行してください。"
```

しかし、`C:\temp\_SSM` は Git リポジトリではない一時作業フォルダだったため、Codex CLI から次のようなエラーが出ました。

```text
Not inside a trusted directory and --skip-git-repo-check was not specified.
```

Codex CLI は通常、信頼できる Git リポジトリ内での実行を前提としています。このプログラムでは、処理ごとに作り直す一時作業フォルダを使いたかったため、Git リポジトリ確認を省略する `--skip-git-repo-check` を追加しました。

次に、Codex が `Prompt.md` を読んで調査や文章生成までは行えるものの、出力ファイルを保存しようとすると、実行環境が read-only のため書き込みを拒否されることがありました。

この用途では、Codex に `Dekigoto.md`、`Output.md`、`Output.html` などの成果物ファイルを作らせる必要があります。そこで、作業ディレクトリ内への書き込みを許可する `--sandbox workspace-write` を追加しました。

結果として、このプログラムでは次の指定を組み合わせています。

- `-C "C:\temp\_SSM"`
  - Codex の作業ディレクトリを指定する
- `exec`
  - Codex を非対話形式で実行し、処理が終わったら終了する
- `--skip-git-repo-check`
  - Git 管理していない一時フォルダでも実行できるようにする
- `--sandbox workspace-write`
  - 作業ディレクトリ内へのファイル作成・更新を許可する

### なぜ Prompt.md を使うのか

短い指示ならコマンドラインへ直接書けますが、実際の作業指示はすぐ長くなります。

このプログラムでは、プロンプトに次のような情報を含めています。

- 入力ファイル
- 出力ファイル
- 作業手順
- 執筆方針
- 史実の扱い
- HTML の仕様
- エラー時の扱い
- 出力後の確認条件

これらをコマンドラインへ直接書くのは扱いにくいため、長い指示は Markdown ファイルとして保存し、コマンド側では「この Prompt.md を読んで実行してください」とだけ伝えます。

## C# 側の役割

C# プログラム側は、主に次の仕事を担当します。

- 作業ディレクトリを用意する
- プロンプトテンプレート内の `{{INPUT_FILE}}` や `{{OUTPUT_FILE}}` を実際のパスに置換する
- 完成した指示を `Prompt.md` として保存する
- Codex CLI を起動する
- Codex が作成したファイルの存在を確認する
- 成果物を保管庫へ移動する
- 使用済みのプロットや素材を削除する
- HTML 一覧用のファイルを更新する

つまり、C# が小説そのものを書いているわけではありません。

C# は工程管理を行い、各工程の実作業を Codex に依頼します。

## ディレクトリ構成

主なソースコードは `HLTConsole/HLTConsole` 配下にあります。

- `Program.cs`
  - 全体の処理フロー
  - 素材補充、執筆、HTML 生成
- `Modules/PromptResource.cs`
  - Codex に渡すプロンプトテンプレート
- `Modules/CodexUtils.cs`
  - `Prompt.md` の作成と Codex CLI の起動
- `Consts.cs`
  - 作業ディレクトリと成果物保存先の定義
- `Commons/SCommon.cs`
  - ファイル操作、外部コマンド実行、乱数、ULID 生成などの共通処理

## 成果物の保存先

現在の設定では、作業ディレクトリと保存先は固定パスです。

```text
作業ディレクトリ:
C:\temp\_SSM

成果物保存先:
C:\home\Chest\20260719_ShortStoryMaker\Storage
```

保存先の下には、次のようなディレクトリが作られます。

```text
Storage
├─ Plot
├─ Material
├─ Story
├─ Illust
└─ HTML
   └─ Data
```

それぞれの役割は次のとおりです。

- `Plot`
  - 未使用のプロット案
- `Material`
  - 未使用の物語素材
- `Story`
  - 完成した短編小説 Markdown
- `Illust`
  - 小説ごとの挿絵 PNG
- `HTML`
  - 一覧ページと補助 JavaScript
- `HTML/Data`
  - 小説ごとの閲覧 HTML

ファイル名には ULID 形式の ID を使っています。小説本文、挿絵、HTML は同じ ID で対応します。

## プロンプトの種類

`PromptResource.cs` には、主に 5 種類のプロンプトがあります。

- `PROMPT_01`
  - 指定年月に起きた歴史的出来事を調査する
- `PROMPT_02`
  - 調査結果から物語素材とプロット案を作る
- `PROMPT_03`
  - プロットと素材から短編小説を書く
- `PROMPT_04`
  - 小説に合う挿絵を生成する
- `PROMPT_05`
  - 小説本文と挿絵を使って HTML ページを作る

このように、ひとつの巨大なプロンプトで全部やらせるのではなく、工程ごとに分けています。

## Codex CLI を使う上での注意

Codex CLI は一般的な Web API とは少し違います。

Web API であれば、HTTP リクエストを送り、JSON で結果を受け取るような形になります。一方、このプログラムでは、Codex CLI をローカルコマンドとして起動し、実際のファイルを読み書きさせています。

そのため、感覚としては次のような使い方です。

```text
入力ファイルを用意する
↓
Prompt.md に作業指示を書く
↓
Codex CLI を起動する
↓
成果物ファイルが作られる
↓
プログラム側で存在確認する
```

AI は通常の関数と違い、毎回完全に同じ結果や動作を返すとは限りません。そのため、このプログラムでは、Codex の実行後に必要な出力ファイルが存在するか確認しています。

また、`--sandbox workspace-write` を指定して Codex に書き込みを許可するため、作業ディレクトリには消されたくないファイルを置かない方が安全です。このプログラムでは専用の作業ディレクトリ `C:\temp\_SSM` を使い、工程ごとに作り直しています。

## 実験結果

パイプラインとしては動きました。

歴史調査、素材化、プロット化、執筆、挿絵生成、HTML 化まで、一連の工程を Codex CLI 経由で自動化できています。生成された短編、挿絵、HTML ページも保存されます。

ただし、最終的な読後感としては、結果は微妙でした。

- ストーリーは期待したほど面白くなかった
- 歴史上の出来事をランダムに選び、さらに素材を組み合わせたことで、物語としてのまとまりが弱くなった可能性がある
- 何も考えず ChatGPT に「物語を書いて」と頼んだ方が、読み物としては面白いものが出るっぽい
- 一方で、ホラーのように断片的な史実、不穏な情景、説明しきれない違和感が活きるジャンルなら相性が良いかもしれない

つまり、このプログラムは「完成度の高い小説生成器」というより、Codex CLI を使って AI 作業を工程化すると何ができて、どこが難しいのかを試した実験記録です。

## 所感

この試みで面白かったのは、Codex CLI をチャット相手ではなく、ファイルを介して仕事をする外部作業員のように扱えることです。

プロンプトをテンプレート化し、C# から変数を埋め込み、`codex exec` で実行することで、AI の作業を自作プログラムの処理工程に組み込めます。

小説生成としては課題が残りましたが、次のような用途には応用できそうです。

- 調査結果を Markdown にまとめる
- 複数資料から素材や論点を抽出する
- プロットや構成案を大量に作る
- テキストから HTML やレポートを生成する
- ファイル成果物を前提にした AI 処理をバッチ化する

ShortStoryMaker は、その実験として作った小さな創作パイプラインです。
